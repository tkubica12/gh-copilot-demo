# Basics

This chapter introduces the fastest way to understand modern GitHub Copilot: start with visible, concrete wins and then connect them to the broader agentic story.

## Why it matters

- Copilot still delivers quick individual wins.
- Model choice, chat, search, and targeted edits remain the on-ramp.
- A short basics chapter makes the transition into agentic delivery feel natural instead of abrupt.

## What to explore first

1. A fast editor or chat win.
2. Codebase understanding on a real repository question.
3. A small multi-file or documentation task.
4. GitHub Spark as an adjacent entry point for natural-language app creation.
5. The transition into agents that can own a bounded engineering task.

## Repository examples

- `src\services\toy\main.py` for a quick inline suggestion or rename
- a repository question such as `Where in my code am I processing messages from Service Bus queues and what is the code doing?`
- a small documentation or multi-file task from `README.md`

## GitHub Spark belongs in the basics story

[GitHub Spark](https://github.com/spark) fits best as a basics-level adjacent capability:

- it is approachable for both technical and less technical participants
- it shows how natural-language app creation sits next to professional engineering workflows
- it creates a clean transition: prototype quickly in Spark, then move serious engineering work into a governed repository with agents, reviews, and workflows

In this repository, Spark acts as an accessible entry point rather than the center of the delivery story.

## What to notice

- Copilot is more than autocomplete, but autocomplete and chat are still useful first steps
- model choice matters when moving from fast assistance to deeper reasoning
- even the basics should end in something concrete: an answer, a diff, or a small working change
