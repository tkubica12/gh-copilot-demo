Jak snížit náklady v 15 minutách

Šetříme tokeny v GitHub Copilot

Od snadných výher po pokročilý context engineering. Cílem není utratit nejméně tokenů, ale získat z každého maximální hodnotu.

Mentální model
00
Co všechno žere tokeny
▾

Náklad neplyne jen z toho, co napíšete do chatu. Sčítá se z mnoha vrstev:

  • Vždy-zapnuté instrukce — AGENTS.md, custom instructions, hooky
  • Vybrané a otevřené soubory v kontextu
  • Historie chatu a shrnutí mezi tahy
  • Definice MCP nástrojů a jejich JSON schémata — i ty, které nepoužijete
  • Výsledky tool callů zopakované jako input v dalším kroku
  • Výstup modelu (output tokeny jsou nejdražší)
  • Retries, subagents, smyčky v agent módu
Cíl je scoped sufficiency — dostatek kontextu, aby Copilot úkol vyřešil správně, ale ani slovo navíc.

Tři typy tokenů — proč na nich záleží:

  • Input — vše, co posíláte modelu poprvé (prompt, kontext, tool results)
  • Cached input — opakovaný prefix, který model už viděl v předchozím tahu téže session. ~10× levnější než fresh input. Proto dlouhá koherentní session může být levnější než restart.
  • Output — to, co model vygeneruje. ~6× dražší než fresh input, ~60× dražší než cached input.

Příklad z reálného aktuálního ceníku (USD za 1M tokenů):

Model Krátký kontext Dlouhý (1M)
Cache Input Output Cache Input Output
gpt-5.5 $0,50 $5,00 $30,00 $1,00 $10,00 $45,00
gpt-5.4 $0,25 $2,50 $15,00 $0,50 $5,00 $22,50
gpt-5.4-mini $0,075 $0,75 $4,50 — — —
gpt-5.4-nano $0,02 $0,20 $1,25 — — —

Mini a nano dlouhý kontext nemají.

Ballpark pravidla (zapamatujte si tahle čísla):

  • Cache : Input : Output ≈ 1 : 10 : 60 — platí napříč celou rodinou. Output je ~6× dražší než fresh input a ~60× dražší než cached. Proto omezit délku výstupu má vždy nejvyšší ROI.
  • Každý tier dolů je ~3–4× levnější: nano → mini → 5.4 → 5.5. Flagship vs nano = 25× rozdíl. Pro extrakci, summarization a triage je nano často dost.
  • Dlouhý kontext (1M) zdraží ~2× input/cache a ~1,5× output. Posílat zbytečně celé repo do 1M okna je nejdražší způsob, jak nic neušetřit.
  • Cache je důvod, proč /compact není zadarmo — nahradíte-li cached history novým summary inputem, platíte 10× víc za stejnou informaci.
Snadné výhry pro každého
01
Použijte Auto model jako default
▾

Pokud si nejste jistí, proč potřebujete premium reasoning model, začněte s Auto.

  • Routuje podle složitosti úkolu, dostupnosti a zdraví systému
  • Nezůstane vám pinnutý drahý model na rutinu
  • 10% sleva na multiplier u placených plánů (Chat, CLI, cloud agent)
  • Manuální override pro architekturu nebo těžký debug zůstává

V praxi to znamená, že pro běžnou práci ušetříte 10 % oproti pinnutí jakéhokoliv konkrétního modelu — a navíc často dostanete levnější model, který úkol zvládne.

02
Explicitní kontext místo širokých promptů
▾
Drahé
Pochop tento repozitář a oprav problém s loginem.
Levné
Soustřeď se na src\auth\login.ts a tests\auth\login.test.ts.
Bug: validateEmail odmítá user+tag@example.com.
Přidej test a oprav jen tohle chování.

Pravidlo: konkrétní cesty k souborům jsou levnější než průzkum celého repozitáře. Široký prompt agenta donutí grepovat, číst, zahazovat — všechno se vám započítá.

Bonus: batchujte související úkoly do jednoho promptu.

Jakmile už zaplatíte za načtení kontextu (AGENTS.md, tool schémata, otevřené soubory), je škoda ho využít jen na jeden drobný úkol. Když agent v jednom tahu opraví bug, přidá test, aktualizuje changelog a doplní docstring, platíte kontext jednou místo čtyřikrát.

V src\auth\login.ts:
1. validateEmail — povol user+tag@example.com
2. přidej test do tests\auth\login.test.ts
3. doplň docstring k funkci
4. zapiš změnu do CHANGELOG.md
Kam až jít? Praktický limit je 3–5 souvisejících úkolů ve stejné oblasti kódu. Nad to agent ztrácí fokus, dělá víc chyb, retries to celé sežerou. A nemíchejte úkoly z různých částí repa do jednoho promptu — to zase nafoukne kontext.
03
Nalezení souborů jako první krok před implementací
▾

Místo „rozumíš tomu celému?" se zeptejte:

Najdi nejmenší množinu souborů, kterou potřebuju pro pochopení event-driven flow.
Vypiš jen cesty a jednu větu proč. Nedělej shrnutí celého repa.

Pak teprve zadejte úkol — s vědomím, co je relevantní. Agent vám vrátí krátký seznam místo dlouhé eseje, a vy z něj vyberete jen to, co skutečně potřebujete v kontextu.

04
Omezte délku výstupu (output tokeny jsou nejdražší)
▾
Drahé
Vysvětli detailně všechno, co jsi změnil.
Levné
Vypiš: změněné soubory, proč, testy. Max 5 odrážek.

„Caveman" styl výstupu pro reporty:

Hotovo. Vypiš: soubory, proč, validace, rizika. Bez úvodu. ≤5 odrážek.
Pozor: nezkracujte bezpečnostní, destruktivní (mazání, force push, DROP) nebo compliance instrukce. Nejednoznačnost stojí víc než ušetřené tokeny.

05
Vkládejte výňatky, ne celé logy — a u UI raději Playwright
▾

U terminálových výstupů screenshot je antipattern — text-as-image musí projít OCR/vision modelem, je dražší a méně přesný než text. Místo 5 000 řádků CI logu pošlete jen relevantní výňatek:

Command: npm test
Exit code: 1
Relevantní chyba:
  TypeError: Cannot read property 'id' of undefined
  at UserService.findById (src/services/user.ts:42)

Posledních 30 řádků:
  ...

U vizuálních otázek (rozložení UI, design feedback, kontrola jak komponenta vypadá v prohlížeči) je obrázek naopak na místě — ale i tady je efektivnější použít Playwright MCP nebo browser canvas:

  • Agent si stránku otevře sám, vyhodnotí ji a vrátí jen zjištění
  • Můžete cílit konkrétní element (selector), ne celou stránku
  • Plyne z toho dynamický kontext — agent může klikat, scrollovat, měřit
  • Místo „tady máš PNG, řekni mi co je špatně" → „otevři localhost:3000, najdi tlačítko Submit, ověř kontrast a velikost"
Pravidlo: text pro text, řízený browser pro vizuál. Statický screenshot až jako poslední možnost.
06
Jazyk a struktura promptu — proč angličtina není dogma
▾

Tokenizer (např. o200k_base, kterým běží GPT modely) je trénovaný hlavně na anglickém textu. Anglická slova se proto často mapují na jeden token, zatímco česká slova s diakritikou se rozsekají na 2–4 tokeny. Stejný počet znaků tedy neznamená stejný počet tokenů.

Ale není to dogma ze tří důvodů:

  1. Strukturovaný formát smaže většinu rozdílu — když jdete z prózy na key/value, čeština je téměř srovnatelná s angličtinou.
  2. Kvalita má přednost před úsporou — pokud myslíte v češtině, anglický prompt může vést k nepřesnostem.
  3. Cena za chybu > cena za tokeny — rework kvůli nejednoznačnému promptu stojí mnohem víc než „drahá" čeština.

Co znamená „normální", „strukturovaný" a „verbose" prompt? Klikněte na příklady níže — všechny tři verze říkají to samé.

Vytvoř POST endpoint /api/users, který validuje povinné pole name a email,
při chybě vrátí 400 a při úspěchu 201 s vytvořeným uživatelem.

Angličtina148311.00×

Tady je rozdíl výrazný — čeština stojí o 61 % víc tokenů, protože diakritika a méně frekventovaná slova jdou přes více tokenů každé.

Závěr: Používejte rodný jazyk, když jde o kvalitu a vyjadřujete nuance. Pro opakované operační prompty (skill instrukce, agent handoffs, šablony) vítězí úsporná angličtina nebo strukturované klíče — bez ohledu na jazyk.

Pokročilé techniky
07
Hierarchický kontext — skills místo obřího AGENTS.md
▾

Always-on instrukce jsou opakovaná daň — platíte je v každém tahu.

Typ kontextuKam patříPravidlo
Always-on (drobné)AGENTS.mdjen fakta, která agent nemůže odvodit
Path-specific.github\instructions\*.instructions.md s applyTonačítá se jen u relevantních souborů
Workflow-specificprompt filesspouští se na vyžádání
Detailní capability.github\skills\progressive reveal — jen když je téma
Live dataMCP serverfetch on demand

Měřený výsledek z mého lab benchmarku:

Velký AGENTS.md vs. malý AGENTS.md + jedna relevantní skill → 68,8 % úspora ve weighted units.

Skill může mít sub-skills nebo odkazy na další kontext — funguje jako lazy-loaded knihovna. Místo monolitu se dokumentace načítá podle potřeby.

Nechte agenta psát kontext pro budoucnost.

Po každé netriviální session máte v paměti agenta destilát: jak je kód poskládaný, kde jsou pasti, jaká jsou rozhodnutí. Místo aby tahle znalost zmizela s koncem sessiony, požádejte agenta, ať ji uloží jako trvalý kontext, který si později načte on sám — nebo váš kolega.

Napiš do .github\skills\auth-flow\SKILL.md
stručný (≤60 řádků) popis auth flow tak, jak jsme ho právě
pochopili. Zaměř se na: vstupní body, klíčové soubory, pasti,
co dělat při změně. Žádná próza, jen seznamy a odkazy.

Co tím získáte:

  • Příští session načte 60 řádků skill místo 30 souborů (řádově menší input)
  • Kolegové dostanou ten samý destilát zadarmo — neplatí znovu za průzkum, který jste už udělali
  • Skill stárne s kódem v repu, ne v hlavách lidí
  • Agent píše vlastní instrukce, podle kterých pak sám pracuje — closing the loop
Pravidlo: pokud jste se v sessioně něco složitě dozvěděli, je to kandidát na trvalý kontext. Repo je sdílená paměť týmu i agentů.
08
MCP progressivně — search → select → fetch
▾

MCP má tři skryté nákladové vrstvy:

  1. Tool definice a JSON schémata načtená do kontextu — i nepoužité
  2. Argumenty tool callu jako output tokeny
  3. Výsledky toolu replayované jako input v dalším kroku

Dobrý workflow:

1. search nebo list kandidáty
2. vyberte jednoho
3. fetch jen detail, co potřebujete pro rozhodnutí
4. shrňte výsledek, než pokračujete
Nepoužívané globální MCP servery stojí tokeny dřív, než je vůbec zavoláte. Držte MCP per-workspace.

A vyhněte se duplikaci — nepřidávejte MCP filesystem server, když klient má vlastní file tools.

09
Deterministické nástroje místo reasoningu přes mnoho tahů
▾

Pokud existuje algoritmus, který úkol vyřeší přesně — nenuťme model dělat to přes několik tahů s pravděpodobností chyby.

Typičtí kandidáti zabalit do skill nebo CLI tool:

  • JSON → XML / CSV konverze
  • Token counting, log slicing, schema validace
  • Extrakce dependency grafu
  • Sort, filter, deduplikace velkých dat
  • Generátory ID, šablony s parametry

Co tím ušetříte: tool-call smyčky, retries, verbose intermediate reasoning, output tokeny, chyby z LLM aritmetiky.

Bonus: tyto CLI nástroje a skills vám může vyvíjet a udržovat sám agent. Stačí říct „tento workflow děláme pořád stejně, zabal ho do skript-backed skill a přidej testy". A na konci session se zeptejte: „Co bych mohl zautomatizovat, abys to příště nemusel řešit promptem?" Agent vám sám navrhne (a často rovnou napíše) CLI tool, který odstraní opakované reasoning přes mnoho tahů.

Skill se script-backed toolem dělá 1 tool call s deterministickým výstupem místo 5 tahů s pravděpodobností chyby.

Session management — /ask, /fork, /resume, /share, /compact
▾
SituaceLepší volba
Stejný koherentní úkol pokračuje/resume — využijte cached context
Vedlejší průzkum, nesmí znečistit main thread/ask nebo /fork
Z užitečného stavu chcete více variant/fork

/compact není rutinní úsporné tlačítko. Cached history je často levnější než ji nahradit fresh summary inputem. Compaktujte, jen když se kontext zreusuje (více forků) nebo je hodně stale.
compact value ≈ reuse_count × removed_stale_context
                − summary_output_tokens
                − fresh_input paid after losing cache
11
Subagents — izolace, ne magická úspora
▾
Paralelní agenti mohou ušetřit wall-clock, ale znásobit input tokeny, pokud čtou stejné soubory.

Použijte subagenta když:

  • Práce je opravdu nezávislá
  • Kontext jde shardovat podle service / file area
  • Stačí levnější model (extrakce, summarization, test triage)
  • Výsledek lze vrátit jako krátké shrnutí

Nepoužívejte když:

  • Všichni agenti potřebují stejný kontext
  • Úkol je sekvenční
  • Coordination overhead dominuje

Handoff small: objective, known facts, exact files, constraints, acceptance criteria, output format. Žádné dlouhé narativní historie.

12
Měření jako součást vývoje skills — CI/CD + self-improvement
▾

Token efficiency není jednorázový audit — je to engineering discipline s feedback loopem.

Měřte při vývoji každé nové skill nebo promptu:

  • Spusťte scénář s baseline (bez skill) a s novou skill
  • Srovnejte: input, output, cached tokens, počet tahů, tool calls, latency, retry rate
  • Používejte weighted units (kombinují cenu input/output/cached) — ne jen raw count
  • Hodnoťte i kvalitu výsledku — úspora s reworkem je ztráta

CI/CD pipeline pro skill changes:

on: pull_request
  paths: ['.github/skills/**', 'AGENTS.md', '.github/prompts/**']

steps:
  - run: copilot-token-lab run --scenarios skills-regression --iterations 3
  - run: copilot-token-lab compare --baseline main --head HEAD
  - fail-if: weighted_units_delta > +10% AND quality_score < baseline
  - comment-pr: report.md
  • PR check, který selže když nová skill zdraží scénář o víc než threshold bez zlepšení kvality
  • Komentář v PR s tabulkou: tokens before/after, $ odhad, kvalita, regression list
  • Trend graf v main branchi — kolik tokenů stojí typický workflow měsíc po měsíci

Self-improving agents (close the loop): po dokončení netriviální session se zeptejte agenta:

Co jsi v této session dělal neefektivně?
Co by ti příště ušetřilo tokeny nebo tahy?
Které části kontextu byly nepoužité?
Navrhni konkrétní úpravu skill / AGENTS.md / prompt template.

V Copilot CLI je na to /chronicle improve — promění recurring friction v krátké instrukce.

OpenTelemetry je ground truth. Copilot CLI exportuje token telemetry přes COPILOT_OTEL_FILE_EXPORTER_PATH. Spojené s automatizovaným harness (např. tools/copilot-token-lab) máte reprodukovatelný benchmark, ne dojem.

Jediná univerzální rada: změřte svůj workload. Cheaper model není automaticky cheaper outcome.
Shrnutí

Snadné

  • Auto model jako default (10% sleva)
  • Pojmenujte přesné soubory a done criteria
  • Nalezení souborů jako první krok
  • Omezte výstup — „max 5 odrážek", „jen diff"
  • Výňatky z logů, ne dumpy; pro UI Playwright
  • Strukturovaný formát smaže rozdíl jazyka

Pokročilé

  • Drobný AGENTS.md + skills + path instructions
  • MCP jako search → select → fetch
  • Deterministické toolery (agent je sám napíše)
  • /ask, /fork, /resume vědomě, /compact opatrně
  • Subagents jen na nezávislou práci
  • Měřte v CI a ptejte se agenta na sebezlepšení

Tokeny nejsou náklad ke škrcení. Jsou investicí. Řešte návratnost, ne spotřebu.
